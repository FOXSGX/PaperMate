# 降重与去 AI 化方案 · 可行性调研报告

> **项目**: PaperMate · 文档处理模块  
> **撰写日期**: 2026-07-18  
> **调研范围**: 文本降重（降低查重率）与去 AI 化（降低 AIGC 检测分）  
> **版本**: v1.0

---

## 1. 问题定义

学术写作中两大痛点：

| 痛点 | 目标 | 典型工具 |
|------|------|----------|
| **降重** | 降低与已发表文献的文字重复率 | 知网查重、Turnitin |
| **去 AI 化** | 降低被 AI 检测工具判定为机器生成的概率 | GPTZero、Originality.ai、ZeroGPT |

二者目标不同，但技术手段有重叠——核心都是改变文本表面形式，同时保持语义不变。

---

## 2. 降重方案分析

### 2.1 基础方法（规则驱动）

| 方法 | 说明 | 效果 | 风险 |
|------|------|------|------|
| 同义词替换 | 词典映射替换高频词 | 中低 | 易被长串匹配检出 |
| 句式变换 | 主动 ↔ 被动、否定前置等 | 中 | 语法错误 |
| 拆分/合并句子 | 长句拆短句、短句合并 | 中 | 流畅度下降 |
| 语序调整 | 状语/定语位置移动 | 低 | 改后变化不明显 |

**评估**: 基础方法对中文降重效果有限（< 10% 重复率降幅），英文略好。适合作为**预处理**，不适合作为主要降重手段。

### 2.2 进阶方法（LLM 驱动）

利用大语言模型进行 paraphrase：

- 输入原文片段 + 降重指令 → LLM 输出改写版本
- 可控制改写幅度（轻度/中度/重度）
- 支持多轮改写 + 对比审阅

**优点**:
- 改写质量高，语法自然
- 可同时处理降重与去 AI 化
- 可针对查重报告中的标红段落定向改写

**缺点**:
- 依赖 LLM API 调用，有延迟和成本
- 过度改写可能改变原意
- 需要设计好 prompt 防止"套话"风格

### 2.3 推荐策略

```
输入文本
   │
   ▼
┌─────────────────────────┐
│ 步骤1: 基础降重          │  ← 规则引擎（rewrite.py）
│  • 同义词替换            │
│  • 句式变换              │
│  • 冗余删除              │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ 步骤2: LLM 深度改写      │  ← API 调用（optional）
│  • 语义保持改写          │
│  • 变化段落结构          │
│  • 补充/精简论证         │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ 步骤3: 人工审阅          │
│  • 逐句对比原文与改写    │
│  • 确认语义未改变        │
└─────────────────────────┘
```

**结论**: 降重功能落地的**最低可行方案**是规则引擎 + LLM 改写。规则引擎可覆盖 10-20% 的降重需求，LLM 改写可覆盖剩余部分。

## 3. 去 AI 化方案分析

### 3.1 AIGC 检测原理

主流检测工具（GPTZero、Originality.ai、ZeroGPT）的核心方法：

1. **困惑度（Perplexity）检测**: AI 生成文本的 token 概率分布更"平滑"，困惑度偏低
2. **突发度（Burstiness）检测**: 人类写作的句子长度和结构变化更大，AI 则趋于均匀
3. **特征模式检测**: 特定句式、过渡词、段落结构的统计特征
4. **水印检测**: 部分 LLM 输出中嵌入的隐式水印

### 3.2 去 AI 化策略

| 策略 | 说明 | 预期效果 |
|------|------|----------|
| 增加句式多样性 | 混合长短句、问句、倒装句 | ★★★★ |
| 引入个性化表达 | 加入口语化连接词、轻微不完美 | ★★★ |
| 人为增加突变度 | 随机插入人类常见的写作"瑕疵" | ★★★ |
| 调整过度词分布 | 减少"此外""然而""值得注意的是"等高频过渡词 | ★★★ |
| 引用与数据混合 | 混入真实数据、引用、脚注 | ★★★ |
| 多模型交叉改写 | 用不同 LLM 交替改写不同段落 | ★★★ |
| 逻辑顺序重排 | 调整论证的先后顺序 | ★★ |

### 3.3 行业现状

| 检测工具 | 国内可用 | 免费额度 | 检测重点 |
|----------|----------|----------|----------|
| GPTZero | ✓ | 有 | 困惑度 + 突发度 |
| Originality.ai | ✓ | 无 | 综合特征 + 水印 |
| ZeroGPT | ✓ | 有 | 简单概率统计 |
| 知网 AIGC 检测 | ✓ | 无 | 疑似 AIGC 标记（灰度） |
| 维普 AIGC 检测 | ✓ | 无 | 疑似 AIGC 标记（灰度） |

> **注意**: 去 AI 化是一个 ongoing arms race。检测工具持续升级，改写策略需不断迭代。不存在"改了就能 100% 过"的方案。

### 3.4 推荐工程方案

```
LLM 改写输出
   │
   ▼
┌──────────────────────────┐
│ 后处理管道                │
│                          │
│ 1. 句式多样性增强          │
│    - 随机插入短句          │
│    - 打乱部分过渡词        │
│                          │
│ 2. 突发度注入             │
│    - 对部分句子增加 or 减少长度 │
│    - 改变连接词使用频率     │
│                          │
│ 3. 统计特征调整           │
│    - 首词分布多样性检查     │
│    - 段落长度方差调整      │
└──────────────────────────┘
   │
   ▼
外部 AIGC 检测器评分
（不断迭代优化参数）
```

## 4. 技术实现路径

### Phase 1（当前）: 规则引擎基础版
- 同义词替换（中文 + 英文）
- 句式变换基础版（主动 ↔ 被动）
- 冗余删除
- 输出改写对比

### Phase 2: LLM 改写集成
- 调用 DeepSeek / Qwen API 实现语义改写
- 支持控制改写幅度参数
- 提供改写对比 UI

### Phase 3: 去 AI 化专项
- 后处理管道（突发度注入、句式多样性）
- 集成 AIGC 检测器反馈循环
- 自动参数调优

### Phase 4: 持续迭代
- 接入更多检测工具做评测
- 用户反馈驱动的改写质量评分
- 针对特定期刊/会议的个性化策略

---

## 5. 风险与局限性

1. **学术伦理**: 降重和去 AI 化的目的是辅助表达，不是造假。需要在 UI 中明确提示用户。
2. **LLM 幻觉**: 基于 LLM 的改写可能引入事实性错误。必须保留原文对照。
3. **检测对抗**: 去 AI 化是猫鼠游戏，无法保证 100% 通过所有检测器。
4. **成本**: LLM API 调用量大的情况下，改写功能可能产生显著费用。

---

## 6. 结论

| 维度 | 可行性 | 说明 |
|------|--------|------|
| 降重（规则驱动） | ✅ 高 | 现版本 rewrite.py 已实现，可满足基础需求 |
| 降重（LLM 驱动） | ✅ 高 | 需要集成 LLM API，后端已有 DeepSeek/Qwen 配置 |
| 去 AI 化（基础） | ✅ 中 | 句式多样性 + 突发度注入可做，效果有限 |
| 去 AI 化（深度） | ⚠️ 待评估 | 需要持续迭代，依赖外部检测器反馈 |

**推荐 MVP**: 先交付降重功能（规则引擎 + LLM 改写集成），去 AI 化作为 Phase 3 专项推进。
